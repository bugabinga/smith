# SM-007: smith-ai/ — LLM Provider Integration

Create the `smith-ai/` crate for LLM provider connections, authentication, and streaming.

## Context

Depends on smith/ (SM-005) only. Does NOT depend on smith-core.
Builds in parallel with smith-core.

**Architecture note:** ProviderEvent and ProviderRequest types live in smith/ (shared).
This crate implements concrete providers that produce ProviderEvent streams.

**Design doc:** `docs/design/AI-CRATE-DESIGN.md`

## Key Design Decisions

1. **Bundled providers.json** — all provider/model data in one JSON file
2. **Auth storage** — `~/.smith/auth.json`
3. **Provider trait** — defined here, uses types from smith/
4. **StreamFn creation** — providers create the StreamFn that smith-core's agent loop calls
5. **Model merge by ID** — default, with `replace_models` flag

### 2. Provider Trait

```rust
use smith::{ProviderEvent, ProviderRequest};
use futures::Stream;
use std::pin::Pin;

/// Provider trait. Each LLM backend implements this.
pub trait Provider: Send + Sync {
    fn id(&self) -> &str;
    fn name(&self) -> &str;

    /// Validate that authentication is configured and non-expired.
    /// Called before the first stream to fail fast on missing/bad credentials.
    fn validate_auth(&self) -> Result<(), ProviderError>;

    fn stream(&self, request: ProviderRequest) -> Pin<Box<dyn Stream<Item = ProviderEvent> + Send>>;
}

/// Create a StreamFn (smith::StreamFn) from a concrete provider.
/// This is what smith-harness passes to the agent loop.
pub fn provider_to_stream_fn(provider: Arc<dyn Provider>) -> Box<smith::StreamFn> {
    Box::new(move |request: ProviderRequest| {
        provider.stream(request)
    })
}
```

### 2a. MuxProvider — Multi-Provider Failover / Rotation

Created by smith-ai after `ModelResolver` determines which provider + account
to use. Wraps one or more `Provider` instances with automatic failover.

```rust
use smith::{ProviderEvent, ProviderRequest, ProviderError};
use futures::Stream;
use std::pin::Pin;
use std::sync::Arc;

/// Multi-provider wrapper with failover and retry.
/// Created from resolved model config (group members or bucket accounts).
pub struct MuxProvider {
    providers: Vec<Arc<dyn Provider>>,
    strategy: smith::FailoverStrategy,
    retry_count: u32,
}

impl MuxProvider {
    pub fn new(
        providers: Vec<Arc<dyn Provider>>,
        strategy: smith::FailoverStrategy,
    ) -> Self {
        Self {
            providers,
            strategy,
            retry_count: 2, // retry non-RateLimit errors twice before failover
        }
    }

    fn stream_with_failover(
        &self,
        request: ProviderRequest,
    ) -> Pin<Box<dyn Stream<Item = ProviderEvent> + Send>> {
        // Try providers in order (or round-robin).
        // On RateLimit: immediate failover to next provider.
        // On other errors: retry up to retry_count, then failover.
        // If all providers exhausted, emit ProviderEvent::Error.
        todo!("implemented in code phase")
    }
}

impl Provider for MuxProvider {
    fn id(&self) -> &str { "mux" }
    fn name(&self) -> &str { "Multiplexed Provider" }
    fn validate_auth(&self) -> Result<(), ProviderError> {
        // Validate all inner providers.
        for p in &self.providers {
            p.validate_auth()?;
        }
        Ok(())
    }
    fn stream(&self, request: ProviderRequest) -> Pin<Box<dyn Stream<Item = ProviderEvent> + Send>> {
        self.stream_with_failover(request)
    }
}
```

**Failover behavior:**
- `RateLimit` -> immediate failover (no retries)
- `AuthFailed`, `Network`, `ServerError` -> retry `retry_count` times with exponential backoff, then failover
- `InvalidRequest`, `ModelNotFound` -> no retry (client error), failover immediately
- `Timeout` -> retry once, then failover
```

### 3. `smith-ai/src/providers.json`

Bundled provider data. Generated by `xtask fetch-providers`.

Each model entry must include metadata consumed by smith-core and smith-harness:
`context_window`, optional `max_output_tokens`, `cost`, and capability flags.
Unknown provider fields are preserved for forward compatibility but ignored by v1.

### 4. `smith-ai/src/model_registry.rs`

```rust
use smith::{ModelCapabilities, ModelCost, ModelMetadata};

pub struct ModelRegistry {
    providers: HashMap<String, ProviderData>,
}

pub struct ProviderData {
    pub id: String,
    pub name: String,
    pub base_url: Option<String>,
    pub auth_types: Vec<String>,
    pub models: HashMap<String, Model>,
}

pub struct Model {
    pub id: String,
    pub display_name: Option<String>,
    /// Canonical model limits/pricing/capabilities.
    /// `context_window` is required for compaction and plugin context usage.
    pub metadata: ModelMetadata,
}

pub struct ProviderConfig {
    pub id: String,
    pub name: Option<String>,
    pub base_url: Option<String>,
    pub auth_types: Vec<String>,
    pub models: Vec<Model>,
    /// False/default: merge models by id. True: replace provider model list.
    pub replace_models: bool,
}

impl ModelRegistry {
    pub fn load() -> Result<Self>;                              // from embedded JSON
    pub fn get(&self, provider: &str, model: &str) -> Option<&Model>;
    pub fn list_models(&self, provider: Option<&str>) -> Vec<&Model>;
    pub fn list_providers(&self) -> Vec<&ProviderData>;
    pub fn register_provider(&mut self, name: &str, config: ProviderConfig);  // merge by default
    pub fn unregister_provider(&mut self, name: &str);
    pub fn unregister_model(&mut self, provider: &str, model_id: &str);
}
```

**Metadata flow:** `ModelRegistry` exposes a concrete model metadata map to
smith-harness; smith-harness passes that map into `ModelResolver::new()`;
`ModelResolver::resolve()` copies `Model.metadata` into `ResolvedModel.metadata`;
smith-core reads `AgentLoopConfig.model_metadata.context_window` for compaction
and `model_metadata.cost` for cost tracking; SM-009 `ExtensionContext` exposes
the same context window to plugins.

### 5. Provider Implementations

- `AnthropicProvider` — anthropic-messages API
- `OpenAIProvider` — openai-completions + openai-responses API
- `GoogleProvider` — google-generative-ai API
- `OpenAICompatProvider` — generic OpenAI-compatible (ollama, MiniMax Token Plan, etc.)

**MiniMax Token Plan details (P12):**
- Base URL: `https://api.minimax.io`
- Auth: bearer API key; Token Plan keys use `sk-cp-` prefix
- Models confirmed: `MiniMax-M2.7`, `MiniMax-M2.7-highspeed`
- Do not use `api.minimax.chat` for Token Plan keys

Each implements `Provider::stream()` → returns `Pin<Box<dyn Stream<Item = ProviderEvent>>>`.

#### Provider Quirks from Prototypes

- Some providers include reasoning/chain-of-thought tokens in the `content` field before the final answer. P12 confirmed this for MiniMax Token Plan. `max_tokens` can be consumed by thinking before useful answer text, so provider adapters must preserve raw content and expose filtered answer text where possible.
- Some provider errors arrive as non-SSE JSON bodies rather than stream chunks; adapters must inspect response body and content type, not only SSE framing.
- Some OpenAI-compatible providers support `stream_options.include_usage`; P12 confirmed MiniMax emits final usage with `prompt + completion = total`. Use this for live token/cost tracking when present.

### 7. Error Module (`smith-ai/src/error.rs`)

Re-exports `ProviderError` and `AuthError` from `smith/` (shared types) and adds conversion impls for external error types.

```rust
use smith::{ProviderError, AuthError};

impl From<reqwest::Error> for ProviderError {
    fn from(err: reqwest::Error) -> Self {
        ProviderError::Network { message: err.to_string() }
    }
}

impl From<oauth2::basic::BasicRequestTokenError<oauth2::reqwest::Error<reqwest::Error>>> for AuthError {
    fn from(err: oauth2::basic::BasicRequestTokenError<oauth2::reqwest::Error<reqwest::Error>>) -> Self {
        AuthError::RefreshFailed { provider: String::new(), reason: err.to_string() }
    }
}
```

*Note:* `ProviderError::Network` stores a `String` message rather than `reqwest::Error` directly because `reqwest::Error` is not `Clone`.
